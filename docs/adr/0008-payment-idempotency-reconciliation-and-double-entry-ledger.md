# ADR-0008 — Idempotency الدفع والمصالحة والدفتر مزدوج القيد

- الحالة: Accepted
- التاريخ: 2026-07-21
- النطاق: E07، مع تأسيس محدود لـE12

## السياق

تدفقات الدفع معرضة لإعادة المحاولة، Webhooks المكررة، فروقات وقت الحركة عن وقت وصولها، اختلاف المبلغ أو العملة، وانتهاء حجز المقعد قبل وصول المصالحة. أي تحديث مباشر لحالة الحجز دون سجل مالي مستقل يمكن أن يكرر التحصيل أو يعيد بيع/تأكيد مقعد بصورة خاطئة.

## القرار

1. يمثل `PaymentIntent` محاولة الدفع المرتبطة بحجز وطريقة ومبلغ وعملة مثبتين.
2. تمثل `PaymentTransaction` الحركة المالية الفعلية، ويمنع تكرارها بواسطة `provider_event_id` أو بصمة مرجع خام.
3. الأوامر التفاعلية تستخدم `Idempotency-Key` ضمن نطاق الحجز أو نية الدفع؛ إعادة الحمولة نفسها تعيد النتيجة، وتغييرها بالمفتاح نفسه يرفض.
4. Webhook الصحيح يُثبت أولًا في `WebhookDelivery` فريد حسب المزود والحدث مع بصمة SHA-256، ثم تعالجه مهمة خلفية؛ هذا يفصل استجابة المزود عن القيود والتذاكر والدفتر.
5. التحويل اليدوي يعتمد `transferred_at` كوقت مالي حقيقي، لا وقت رفع الإثبات أو مراجعته.
6. نجاح الدفع لا يعني تلقائيًا إعادة تأكيد الحجز: يعاد قفل الحجز وفحص الموعد والمقاعد. عند فقدان المقعد يسجل المال وتفتح `PaymentReconciliationCase` للاسترداد أو البديل.
7. اختلاف المبلغ أو العملة يسجل حدث المزود كحركة Failed لمنع إعادة معالجته، ويبقي الدفع غير ناجح ويفتح مصالحة.
8. كل حركة ناجحة تنشئ `LedgerEntry` بطرفين أو أكثر متوازنين قبل تغيير الحالة التجارية.
9. يفرض PostgreSQL توازن القيد عند Commit بواسطة Deferred Constraint Trigger، إضافة إلى التحقق التطبيقي.
10. القيود والتصحيحات المالية اللاحقة تكون بأحداث عكسية أو Adjustments، ولا تعدل التاريخ المالي بصمت.
11. تنشأ العمولة المتوقعة من Snapshot الحجز؛ دورة كسبها وتسويتها تغلق ضمن E12.

## النتائج

- يمكن إعادة Webhook أو طلب المستخدم بأمان دون Inbox أو حركة أو قيد مكرر.
- لا يعتمد رد المزود على اكتمال المعالجة المالية؛ الفشل محفوظ كحالة تشغيلية قابلة للتشخيص وإعادة المعالجة المحكومة.
- يبقى وقت الدفع الحقيقي محفوظًا حتى عند التأخر في المراجعة.
- لا يعاد مقعد مباع بسبب دفع متأخر.
- توجد حالة تشغيلية صريحة لكل اختلاف يحتاج استردادًا أو بديلًا أو مراجعة مبلغ/عملة.
- PostgreSQL يبقى مصدر الحقيقة النهائي لسلامة الدفتر؛ SQLite تستخدم فقط للاختبارات غير الخاصة بالقيد المؤجل.
