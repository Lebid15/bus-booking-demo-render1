# ADR-0006 — تثبيت الحجز وSnapshots ورمز الإدارة

- الحالة: Accepted
- التاريخ: 2026-07-21
- النطاق: E06/E07، مع شريحة تأسيسية من E08

## السياق

يتحول Seat Hold قصير العمر إلى حجز رئيسي دائم. يجب منع السباقات، وعدم إعادة حساب السعر أو السياسات بعد الإنشاء، وإصدار رمز إدارة لا يخزن كنص صريح.

## القرار

1. تقفل الخدمة الرحلة والـHolds والمقاعد المطلوبة داخل معاملة قاعدة بيانات واحدة.
2. يحفظ كل Hold `quote_version` و`quote_snapshot` وقت إنشائه، ثم يرفض التأكيد إذا تغير إصدار الرحلة.
3. ينشأ `Booking` و`BookingPassenger` و`SeatAssignment` وتتحول الـHolds إلى `consumed` في المعاملة نفسها.
4. تطبق قاعدة الجنس فقط على علاقات `same_unit`. اختلاف الجنس داخل الحجز نفسه مسموح، أما الحجز المستقل فيرفض دون كشف جنس أو هوية الراكب المجاور.
5. ينسخ الحجز السعر والسياسات والعمولة والمهلة وإصدارات الموافقة في حقول Snapshot الخاصة به.
6. ينشأ PNR عشوائي مستقل عن المفتاح الداخلي.
7. رمز الإدارة مشتق بـHMAC من معرف الحجز ومفتاح سري مستقل، ويخزن فقط SHA-256 له. لا يخزن النص الصريح داخل الحجز أو سجل Idempotency؛ يعاد توليده بصورة حتمية عند إعادة الطلب المطابق.
8. يسجل إنشاء الحجز في Audit وTransactional Outbox دون تنفيذ إشعار خارجي داخل المعاملة.

## النتائج

- إعادة طلب Idempotency تعيد الحجز نفسه ورمز الإدارة نفسه.
- تغيير إعدادات الرحلة أو المكتب لاحقًا لا يغير الحجز التاريخي.
- قيد قاعدة البيانات يبقى خط الدفاع الأخير ضد إسناد المقعد مرتين.
- تدوير مفتاح رمز الإدارة يتطلب خطة انتقال لأن الرموز السابقة تعتمد عليه.
- حماية مجموعة الطفل/المرافق تحفظ Snapshot للمجاورة، بينما التنفيذ الكامل لإعادة توزيع البولمان يبقى ضمن E11.
