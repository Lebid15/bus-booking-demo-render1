# تدفقات المستخدم

## تدفق الحجز كضيف

1. البحث ببيانات صحيحة.
2. فتح نتيجة وقراءة السعر والسياسة.
3. إنشاء Seat Hold عند أول اختيار مقعد.
4. توزيع الركاب على المقاعد والتحقق من المجاورة والجنس.
5. إدخال البيانات والتحقق منها.
6. مراجعة Snapshot والسعر النهائي.
7. اختيار الدفع وإنشاء الحجز.
8. عند الدفع في المكتب/التحويل: تظهر المهلة والتعليمات والحالة غير المدفوعة.
9. عند الدفع الإلكتروني: ينتقل إلى مزود آمن ثم يعود إلى نتيجة لا تعتمد على Redirect وحده؛ Webhook مصدر التأكيد.
10. يحصل على PNR ورمز إدارة وتذكرة عند استيفاء شروط الإصدار.

## تدفق حجز المكتب

رحلة اليوم ← خريطة المقاعد ← اختيار المقعد ← البحث عن راكب متكرر أو إدخال جديد ← تحديد طريقة الدفع ← تأكيد ← إيصال/تذكرة. يجب ألا يتجاوز التدفق الاعتيادي شاشتين أو Drawer واحدًا بعد اختيار الرحلة.

## تدفق تغيير البولمان

اختيار رحلة ← اختيار بولمان جديد ← محاكاة إعادة التوزيع ← قائمة تعارضات ← حل يدوي مخول عند الضرورة ← اعتماد ← إبطال تذاكر قديمة ← إصدار جديد ← إشعار المتأثرين ← حفظ الأدلة.

## تدفق رفض الصعود

فتح حالة P1 تلقائيًا/يدويًا ← تجميد No-show وإعادة البيع ← تحقق من الحجز والدفع والهوية والQR ← قرار فوري: صعود/مقعد مكافئ/رحلة بديلة/استرداد وتعويض ← توثيق السبب وإغلاق بعد قبول الحل.

## تدفق الاسترداد

طلب أو إجراء مسبب ← Quote حسب Snapshot ← فحص نزاع/Chargeback سابق ← موافقات حسب العتبة ← إرسال للمزود أو تسجيل نقدي/يدوي ← Ledger Event ← تحديث الحالة ← إشعار ومصالحة.
