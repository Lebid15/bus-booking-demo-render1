# سلطة الوثائق وترتيب المرجعية

## الهدف

تمنع هذه الوثيقة تضارب القرارات أثناء البرمجة والتشغيل. لا يجوز لوكيل البرمجة أو فريق التصميم أو التشغيل اختيار تفسير بديل عندما توجد قاعدة أعلى منه.

## ترتيب المرجعية

1. القانون المحلي والالتزامات التنظيمية الملزمة بعد مراجعتها قانونيًا.
2. قواعد سلامة المال والمقاعد وعزل المكاتب والخصوصية.
3. `MASTER_PROJECT_SPECIFICATION_AR.md`.
4. مخطط قاعدة البيانات وقيوده في `04_DOMAIN`.
5. آلات الحالات في `05_STATE_MACHINES`.
6. دفتر الأستاذ والأحداث المالية في `07_PAYMENTS_FINANCE`.
7. عقد API في `11_API/openapi.yaml`.
8. الصلاحيات والإعدادات والسياسات التشغيلية.
9. مواصفات الشاشات والتصميم.
10. نصوص الواجهة والاقتراحات البصرية.

## قواعد الحسم

- عند تعارض الواجهة مع قيد قاعدة البيانات، يُعد القيد هو الصحيح وتُعدل الواجهة.
- عند تعارض سياسة مكتب مع حد إلزامي للمنصة، يُرفض إعداد المكتب.
- عند تعارض إعداد جديد مع Snapshot حجز قائم، يبقى Snapshot الحجز نافذًا.
- لا تُعدل القيود المالية أو حالات الحجز من أجل تبسيط الواجهة.
- أي تغيير في البنود 1–7 يحتاج Change Request موثقًا واختبارات ترحيل وعدم رجعية.

## تصنيف الوثائق

- **ملزمة للتنفيذ:** المجال، الحالات، API، المال، الصلاحيات، الأمن، القبول.
- **ملزمة للتشغيل:** السياسات، إجراءات المكتب، الدعم، الإطلاق.
- **إرشادية قابلة للتحسين:** الهوية البصرية الدقيقة والرسوم والأيقونات ما لم تؤثر في قابلية الاستخدام أو الوصول.
- **مؤجلة:** العناصر المسجلة صراحة في النطاق المؤجل.
